# NimHunter v2 — Transpilation Signature Framework

NimHunter is a multi-layered detection framework specifically designed to detect **Nim-compiled malicious binaries**, even when they are subjected to aggressive polymorphism and metamorphism. 

This tool implements the research thesis: *"The Transpilation Signature: A Multi-Layered Detection Framework for Nim-Compiled Malicious Binaries"*.

## Why This Works Against Polymorphic Malware

A metamorphic engine typically mutates user-defined logic. However, it is generally incapable of rewriting the entire Nim runtime library or the mandatory initialisation sequence (e.g., `NimMain`, `nimRegisterGlobalMarker`, ORC garbage collector). By tracking the **computational asymmetry** generated by the Nim to C transpilation pipeline, NimHunter identifies structural and behavioral invariants that attackers *cannot* remove without breaking the malware itself.

## 🚀 Installation & Build

NimHunter is fully cross-platform and will run on **macOS**, **Kali Linux**, and **Windows**.

### 1. Install Dependencies
You need the Nim compiler and (optionally) YARA.

**Kali Linux / Debian:**
```bash
sudo apt update
sudo apt install nim yara gcc-mingw-w64-x86-64
```

**macOS:**
```bash
brew install nim yara mingw-w64
```

**Windows:**
```powershell
choco install nim yara
# OR 
scoop install nim yara
```

### 2. Build NimHunter
Clone the repository and run the standard Nimble build:

```bash
git clone https://github.com/Mr-Malman/NimHunter.git
cd NimHunter
nimble build
```
This produces the `nimhunter` executable (`nimhunter.exe` on Windows).

## 🛠️ Usage Examples

NimHunter supports single-file scanning, batch directory scanning, and JSON output for pipeline integration.

### Single File Scan
```bash
./nimhunter data/samples/nim_orc_malware.exe
```

### Batch Scanning (Directory)
```bash
./nimhunter --batch data/samples/
```

### JSON Output (Pipeline Integration)
```bash
./nimhunter --json data/samples/nim_orc_malware.exe | jq
```

### Options
- `--rules <path/to/main.yar>`: Specify a custom YARA orchestrator rule file.
- `--no-color`: Disable ANSI color output (useful for piping to text files).
- `--generate-trainer`: Emit the Python script used to train the ML ensemble model.

## 🧠 Interpreting the Output

NimHunter processes binaries through 4 layers, accumulating a score out of 100.

| Layer | Max Score | What it tracks | Resilience |
|---|---|---|---|
| **Layer 1: YARA** | 25 pts | Static strings, known malware signatures, XOR stubs | Low (Strings can be stripped/encrypted) |
| **Layer 2: Structural** | 75 pts | `NimMain` hierarchy, GC hooks, calling density, offensive libraries | **High** (Mandated by compiler transpilation) |
| **Layer 3: Poly/Meta** | Info | Evaluates the detected structural indicators to measure evasion risk | Contextual |
| **Layer 4: ML Heuristic** | 10 pts | Weighted ensemble prediction | Moderate |

### Evasion Risk Rating (Poly/Meta Layer)
If you analyze a highly obfuscated or polymorphic binary, Layer 3 provides the **Mutation Resilience Report**:
- **LOW Risk**: The malware relies on structural invariants (like the ORC GC motif) that the mutation engine *cannot* hide. You have definitively attributed the binary to the Nim compiler.
- **HIGH Risk**: The detection relies only on metadata (like `_TM` strings or `@m` module paths) which an advanced obfuscator could strip in a future mutation.

### What is an "Offensive Library"?
NimHunter flags the presence of specific third-party packages in mangled names (e.g., `winim`, `nimprotect`, `strenc`, `ptr_math`). Legitimate Nim software rarely pairs direct Windows API wrapping (`winim`) with strong string encryption (`strenc`); their presence heavily correlates with Nim-based malware.

## 🧬 Reverse Engineering Workflow

If you want to manually reverse engineer the artifacts NimHunter flags:

1. **Verify the string table:** NimHunter automatically attempts to reconstruct the `_TM` string objects from the `.rdata` section. These often contain unobfuscated C2 URLs.
2. **Find the heart of the payload:** Open your disassembler (IDA/Ghidra/Radare2). Search for cross-references (Xrefs) to `nimRegisterGlobalMarker`. The function containing the highest density of these calls is almost invariably `NimMainModule` — where the attacker's main logic resides.
3. **Generate Sample Binaries:** Use the included `data/make_samples.nim` to cross-compile clean test payloads for benchmark comparisons.
