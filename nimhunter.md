# NimHunter v2: Technical Tool Profile

**Tool Name:** NimHunter v2  
**Primary Purpose:** A multi-layered, cross-platform detection framework for identifying and attributing Nim-compiled malicious binaries.  
**Thesis Alignment:** "The Transpilation Signature: A Multi-Layered Detection Framework for Nim-Compiled Malicious Binaries"

---

## 1. Executive Summary
NimHunter v2 is a specialized reverse-engineering and detection tool engineered to identify malware written in the Nim programming language. As threat actors increasingly adopt Nim for its cross-platform capabilities, C-level performance, and ability to bypass traditional AV/EDR signatures, NimHunter counters this by shifting detection from behavioral heuristics to **architectural invariants**. 

By exploiting the "computational asymmetry" of the Nim-to-C transpilation pipeline, NimHunter identifies the immutable artifacts generated by the Nim compiler—referred to as the "Transpilation Signature."

## 2. Core Operational Concept: Computational Asymmetry
Metamorphic and polymorphic malware engines mutate user-defined logic to evade signature-based detection. However, Nim malware is uniquely constrained: a mutation engine cannot rewrite or scramble the Nim Runtime Library (RTL) or its mandatory initialization sequence without breaking the executable. 

NimHunter evaluates this asymmetry. Even if an attacker uses XOR encryption on their payload or permutes code blocks, the structural foundation of the Nim language (String structs, Garbage Collector hooks, Initialization hierarchies) must remain intact. NimHunter's detection logic anchors onto these mandatory structures.

## 3. The Four-Layer Detection Architecture

NimHunter processes binaries through a deterministic, four-layer pipeline, accumulating a definitive "maliciousness score" (0-100).

### Layer 1: Static Artifact Layer (YARA)
- **Mechanism:** Scans the unmapped binary `.rdata` and `.data` sections for known Nim substrings, standard library paths, and known evasive/offensive toolkit signatures (e.g., Winim wrappers, XOR decryptor stubs).
- **Limitation Mitigated:** While static strings can be stripped or obfuscated, this layer serves as a fast, initial triage mechanism.

### Layer 2: Structural Invariant Layer
- **Mechanism:** The core of NimHunter. It parses the executable layout and reconstructs heuristic footprints of the transpiler's control flow graph.
- **Invariants Tracked (10 Components):**
  1. **NimMain Hierarchy:** Detects the required `tmainCRTStartup` -> `NimMain` -> `NimMainInner` -> `NimMainModule` cascade.
  2. **GC Marker Clustering:** Locates dense cross-references to `nimRegisterGlobalMarker`, anchoring the location of the main malware logic.
  3. **Module Encoding (`@m_`):** Identifies Nim's unique package representation scheme.
  4. **_TM String Structs:** Recovers initialized string objects by identifying length+payload structs, often revealing obfuscated Command & Control (C2) URLs.
  5. **sysFatal Safety Checks:** Identifies language-mandated bounds/division safety checks.
  6. **ORC Tri-Color Motif:** Detects the highly complex trial-deletion state machine of Nim's ORC garbage collector.
  7. **ARC Hooks:** Detects `=copy` and `=destroy` memory management hooks injected by the compiler.
  8. **Foreign Thread GC:** Identifies `setupForeignThreadGc`, a definitive indicator that Nim is being executed inside an injected thread (e.g., a shellcode loader).
  9. **Call Density:** Measures abnormal `0xE8` (CALL) opcode density characteristic of transpiled C code.
  10. **Offensive Libraries:** Identifies high-risk third-party imports (e.g., `winim`, `strenc`, `ptr_math`, `nimprotect`).

### Layer 3: Polymorphic & Metamorphic Resilience Engine
- **Mechanism:** Rather than just detecting malware, this layer *evaluates the detection itself*. It categorizes the found artifacts into "Mutation Sensitive" (can be stripped by an attacker) and "Mutation Resistant" (mandated by the compiler).
- **Output:** Generates an "Evasion Risk Level". If detection relies on the ORC GC motif or NimMain hierarchy (Resistant), the binary is definitively categorized as Nim, regardless of the metamorphic engine used.

### Layer 4: Machine Learning Heuristic
- **Mechanism:** Normalizes the scores from the structural components and PE metadata (entropy, section counts) into a 17-dimensional feature vector.
- **Output:** Passes the vector through an ensemble inference engine to provide a final probabilistic confidence score, smoothing out edge cases where binaries are heavily stripped.

## 4. Reverse Engineering Utility (Forensics)
Beyond detection, NimHunter acts as an automated triage tool for reverse engineers:
- **String Recovery:** It automatically reconstructs Nim string tables from temporary (`_TM`) structures, bypassing basic string obfuscation.
- **Attribution (Demangling):** Nim's name mangling process embeds absolute file paths from the developer's machine. NimHunter decodes these, extracting the attacker's package names, toolkit versions, and occasionally their local filesystem usernames, providing critical Threat Intelligence (CTI).

## 5. Codebase Architecture & Implementation Materials

To support reproducibility and formal academic review, the NimHunter tool was developed using the following stack, architectures, and libraries:

### Technology Stack & Dependencies
- **Core Language:** Nim (v2.0.0+). Chosen for its ability to compile directly to C, allowing NimHunter to seamlessly interact with raw byte buffers and memory addresses without the overhead of a virtual machine.
- **Compiler Backends:** GCC / MinGW-w64. Used for cross-compiling the detection engine to target Windows (`.exe`), Linux (`.elf`), and macOS native binaries.
- **External Scanning Engine:** YARA (v4.3+). Executed via cross-platform shell wrappers to apply static heuristic rulesets (`rules/main.yar`).
- **Standard Libraries Utilized:** 
  - `strutils`, `strformat` (String manipulation and demangling).
  - `math` (Entropy calculation via Shannon's formula).
  - `os`, `sequtils`, `parseopt` (Pipeline orchestration and CLI argument parsing).

### Internal Module Structure
The framework is strictly decoupled into distinct functional domains:

1. **Orchestrator (`src/nimhunter.nim`):** The main ingestion pipeline. It parses CLI arguments, ingests the target binary into byte sequences, and cascades the data through the detection layers.
2. **Analysis Engine (`src/analyzer/`):**
   - **`pe_parser.nim`:** A custom-built, lightweight PE (Portable Executable) parser. Instead of relying on heavy third-party parsing monoliths, this module extracts ONLY what is required for detection: 
     - Entry Point RVA calculation.
     - Section table bounds to compute Shannon entropy per memory region.
     - Import Directory extraction (to look for suspicious DLLs).
   - **`structural.nim`:** The core logic engine that scans the raw byte arrays for the 10 invariant heuristics (e.g., identifying the Hex byte sequence `E8 ?? ?? ?? ??` mapping to relative CALLs for control flow density).
   - **`demangler.nim`:** Implements string slicing logic to decode Nim's proprietary `@m` and `pureZ` package namespace encodings into human-readable forensic paths.
3. **Detection Modules (`src/detectors/`):**
   - **`poly_meta_resilience.nim`:** Analyzes the `DetectionResult` object from the structural layer to calculate the overall Evasion Risk Level.
   - **`ml_engine.nim`:** Normalizes structural scores (e.g., `(score / maxScore)`) into a 17-dimensional `float` vector for algorithmic evaluation.
   - **`yara_engine.nim`:** Orchestrates the child process execution of the YARA binary and parses the standard output back into the Nim application state.

### Key Algorithmic Logic
- **Shannon Entropy Calculation:** Used to detect packing or XOR encryption. NimHunter iterates through the byte frequency of the `.text` and `.data` sections to calculate entropy using the standard mathematical formula: `H(X) = -Σ(P(x) * log2(P(x)))`. High entropy (> 7.0) often signals encrypted payloads.
- **Sliding Window Byte-Scanning:** Rather than full disassembly (which is slow and easily defeated by anti-disassembly tricks), NimHunter treats the executable as a raw byte array. It uses efficient Boyer-Moore string matching and targeted hex-sequence sliding windows to map out the topological clusters of `nimRegisterGlobalMarker` and GC initialization sequences.

## 6. Summary for Academic Inclusion
NimHunter practically demonstrates the theoretical defensive advantages outlined in "The Transpilation Signature." By proving that modern, high-level compiled languages leave indelible topological marks on their resulting binaries, NimHunter establishes a blueprint for future detection systems targeting emerging languages like Rust, Go, and V.
